# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository purpose

This is a template repository for bootstrapping new full-stack applications. It has three independently versioned and deployed components: `backend/` (FastAPI), `frontend/` (Vite/React), and `scripts/` (build + deployment tooling). They don't share a release cadence — see "Independent versioning" below. Its backend is wired to a separate, shared authorization service at `auth.withfbraun.com` — see "Authorization" below, which is the largest and most actively-evolving part of this template.

## Workflow requirements

- **Commit after every change.** Once a change (feature, fix, config edit) is complete and working, create a git commit for it before moving on to the next one.
- **Every backend change needs test coverage.** After adding or modifying anything under `backend/app/`, create or update the matching test(s) under `backend/tests/` in the same change — do not leave backend code changes untested.
- **Every frontend change must pass `npm run build`** (`tsc` + `vite build`) before being considered done; UI-visible changes should also be exercised in a browser, not just typechecked.
- **New source files get the standard header** — see "File headers" below for the exact text and placement rules.
- **Every change updates release news.** Prepend a short, user-facing bullet describing the change to `backend/release_news.json`'s `unreleased` array (backend changes) or `frontend/public/release_news.json`'s (frontend changes), in the same commit as the change itself. `scripts/build.py` seals these into a dated, versioned release at build time — see Deployment below.

## Commands

### Backend (`backend/`)
Runs from a local venv at `backend/.venv` (Python 3.14). All commands below assume cwd = `backend/`.
- Dev server: `.venv/Scripts/python.exe -m uvicorn app.main:app --reload` (port 8000)
- All tests: `.venv/Scripts/python.exe -m pytest`
- Single file: `.venv/Scripts/python.exe -m pytest tests/test_auth_router.py`
- Single test: `.venv/Scripts/python.exe -m pytest tests/test_auth_client.py::test_login_consent_required_does_not_raise`
- New migration: `.venv/Scripts/python.exe -m alembic revision --autogenerate -m "message"`
- Apply migrations: `.venv/Scripts/python.exe -m alembic upgrade head`
- After adding a dependency: install it, then re-freeze with `pip freeze > requirements.txt` (the file is a full freeze of the venv, not a hand-maintained list)

### Frontend (`frontend/`)
- Dev server: `npm run dev` (port 5173)
- Build (typecheck + bundle): `npm run build` — runs `tsc` (noEmit type-check) then `vite build`
- Preview production build: `npm run preview`

## Architecture

### Backend: config, routing, DB, logging, email
- All settings load through `app/config.py`'s `Settings` (pydantic-settings), sourced from `backend/.env` (see `.env.example`). `BASE_DIR` there anchors all relative paths (env file, log dir) to `backend/` regardless of the process's cwd. `get_settings()` is `@lru_cache`d — a running process never picks up `.env` changes without a restart (`uvicorn --reload` only watches `.py` files, not `.env`).
- All REST routes are mounted under the `/api` prefix via `app/api/router.py`, which aggregates sub-routers (`app/api/public/version.py`, `app/api/auth/router.py`, `app/api/account/router.py`). Add new endpoints as their own `APIRouter` and wire them into `api_router` there, rather than adding routes directly in `main.py`.
- DB access is async-only: SQLAlchemy's async engine + `asyncpg` (the `DATABASE_URL` scheme is `postgresql+asyncpg://`, which has no sync counterpart — don't reach for `psycopg2`-style sync sessions). `app/models/base.py` has an empty `Base`; `app/models/user.py`'s `User` is the only concrete model so far (see Authorization below) — holds just the identity link (`auth_sub`) plus fields genuinely local to this app (currently one illustrative example, `nickname` — see the local account resource below). Identity fields (email, name, role, ...) stay owned by the auth service and are read fresh from the JWT / `GET /api/auth/me`, never duplicated here.
- Alembic is wired for the async engine (`alembic init -t async`). `alembic/env.py` pulls the connection string from `Settings.database_url`, not from the static placeholder in `alembic.ini`, and sets `prepend_sys_path = %(here)s` so `import app` resolves regardless of invocation directory. Any new model must be imported in `alembic/env.py` (see the `from app.models.user import User` line there) or autogenerate won't see its table.
- Logging is configured once at import time in `main.py` via `configure_logging()` (`app/logging_config.py`) — a `TimedRotatingFileHandler` (fixed daily rotation, `when="midnight"`) + console handler attached to the root logger. Settings field names match `.env.example`'s `LOGGING_*` keys exactly: `logging_level`, `logging_dir`, `logging_filename`, `logging_days_history` (backup count), `logging_message_format`. Note `.env`/`.env.example`'s format string's `\t` is a literal backslash-t (two characters), not an actual tab — `.env` files don't interpret string escapes the way Python source does, and `Settings` passes the value straight to `logging.Formatter()` unmodified. That's a property of the configured value, not a bug to fix in code; change it in `.env` (to a real tab or something else) if the literal `\t` in log output isn't wanted.
- Email goes through `app/services/email.py`'s `send_email()`, built on `fastapi-mail`. Connection defaults come from `Settings` (`MAIL_*` in `.env`), but every `ConnectionConfig` field can be overridden per call via kwargs (e.g. a one-off `MAIL_FROM`). `MAIL_SUPPRESS_SEND=true` still builds and dispatches the message internally (so tests can assert on it via `FastMail(...).record_messages()`), it just skips opening a real SMTP connection.

### Backend: authorization (auth.withfbraun.com)
This template doesn't implement its own user auth — it's a **backend-for-frontend proxy** in front of a separate, shared auth service. The browser only ever talks to this app's own `/api/auth/*`; only this backend holds the `X-Api-Key` secret and talks to `auth.withfbraun.com` directly. That service's OpenAPI spec (`https://auth.withfbraun.com/openapi.json`) is the authoritative contract and **is actively evolving** — two real discrepancies have already been found between it/its manual and live behavior (an endpoint that didn't exist yet, a response shape that didn't match the docs). Re-fetch and diff against it before trusting any assumption about paths or schemas; don't rely on memory of a previous session's findings.

- `app/services/auth_client.py`'s `AuthClient` (module-level singleton `auth_client`, closed in `main.py`'s `lifespan` alongside the DB `engine`) wraps every call to the auth service: `register`, `login`, `refresh`, `logout`, `get_me`, `update_me`, `get_group_attributes`, `update_group_attributes`, `get_public_key`, `get_group_info`, `get_consent_info`, `grant_consent`, `reject_consent`. **`X-Api-Key` is mandatory on every single method** (`_require_api_key()` raises `RuntimeError` immediately if it's empty, rather than silently sending an empty header — the auth service treats a missing/empty key as "no application context" and skips its consent check entirely, which is a real security-relevant footgun, not just a style choice). `login()` is the one method that returns the raw `httpx.Response` instead of raising on non-2xx: its caller needs to distinguish "200 with tokens" from "403 consent_required" itself.
- `app/security/jwt.py` verifies access tokens locally (RS256) against a public key fetched once from `GET /api/auth/public-key` and cached in a module-level variable guarded by an `asyncio.Lock` — deliberately not `functools.lru_cache` on an `async def`, which would cache the coroutine object rather than its awaited result. Uses **PyJWT**, not the auth service's own manual's suggested `python-jose` (python-jose has a disclosed, unpatched `alg=none` bypass, CVE-2025-61152, and is broadly unmaintained — FastAPI's own docs have moved to PyJWT too).
- `app/api/deps.py`'s `get_current_user` composes JWT verification with the local DB: resolves (or creates, via `INSERT ... ON CONFLICT (auth_sub) DO NOTHING` + re-select — not a naive check-then-insert, which would race two near-simultaneous first requests from the same new user) the `User` row for the verified `sub` claim.
- `app/api/auth/router.py` exposes `register` / `login` / `refresh` / `logout` / `me` (`GET`) / `update_me` (`PATCH /me`) / `get_group_attributes` + `update_group_attributes` (`GET`/`PATCH /me/group-attributes/{group_id}`) / `group-info` / `consent/{id}/grant` / `consent/{id}/reject`, matching exactly what `frontend/src/context/AuthContext.tsx`, `Account.tsx` and `api/client.ts` call. Non-obvious points:
  - `GET /me`'s upstream response (`MeResponse`) always has `consent_required` + `application_group_id`; `user` is populated only once consent is granted. Our own `/me` converts `consent_required: true` into a 403 with that same `{consent_required, application_group_id}` shape — matching `/login`'s own 403 convention, so the frontend has one code path for both (this also covers entry points that skip `/login`'s check, e.g. Google OAuth). On success, `application_group_id` is merged into the flat response too (`{**user, application_group_id}`), so the frontend can call the group-attributes endpoints below without a second round trip; `PATCH /me` doesn't repeat this merge since that id never changes on a profile edit. `result.get("user") or result` is a defensive fallback for an earlier observed state where a granted response had no wrapper at all — cheap insurance, shouldn't trigger against the currently-published schema.
  - `PATCH /me/group-attributes/{group_id}` **replaces the entire upstream `user_data` object — it is not a partial merge.** Callers (see `Account.tsx` below) must fetch the current value first and send the full merged object. The backend treats `user_data` as a fully opaque JSON blob; the shape it *should* have is defined client-side (`public/config.json`, see below), not here.
  - The auth service's own `UserGroupAttributesPublic` schema documents itself as "user_data only, system_data is hidden" — `system_data` is not reachable through any endpoint this app has credentials for, at any layer, confirmed against the live OpenAPI spec. Don't build backend support for it.
  - Consent is resolved through this template's **own** `/consent` page (`frontend/src/pages/Consent.tsx`), not the auth service's hosted one — it calls `GET /api/auth/group-info` (X-Api-Key only, no user token) to render the scopes, then `POST /api/auth/consent/{id}/grant` or `.../reject`.
- `app/api/account/` is a **separate, local-only** resource (`GET`/`PATCH /api/account/me`) for fields this app owns outright (currently just `nickname`). It depends only on `get_current_user` and the local DB session, and deliberately has **no import of `auth_client` anywhere in the package** — that absence is itself the enforcement that editing local account data can never reach the auth service. When adding real app-specific fields, keep this separation: local data goes through `app/api/account/`, identity/profile data goes through the auth-service-proxying endpoints above.
- Testing patterns specific to this layer (see `tests/conftest.py`): `generate_rsa_keypair()` / `make_access_token(sub=..., **claims)` for signing test JWTs without touching the real service; `respx` mocks every upstream HTTP call (no test should ever hit `auth.withfbraun.com`); route tests that need both a real JWT dependency chain *and* the DB (e.g. `/me`) use `httpx.AsyncClient(transport=ASGITransport(...))` instead of the `client` (`TestClient`) fixture — `TestClient` drives the ASGI app from a separate thread with its own event loop, which conflicts with `db_session`'s connection (bound to the test's own loop); `ASGITransport` keeps everything on one loop. `pytest.ini` also sets a **session-scoped** asyncio event loop for the same underlying reason: `app/database.py`'s `engine` is a module-level singleton whose connection pool can't be split across per-test loops.

### Cross-cutting: backend and frontend version independently
- `backend/version.json` and `frontend/public/version.json` are separate, independently-maintained version markers — there is no shared version number between the two components.
- `GET /api/public/version` reads and returns `backend/version.json` verbatim.
- The frontend's `/version` page (`frontend/src/pages/Version.tsx`) fetches both that endpoint and its own `/version.json`, and flags a mismatch whenever their major.minor versions differ, using the shared `parseMajorMinor`/`compareVersions` helpers in `src/utils/version.ts`.
- `scripts/build.py` packages and tags backend/frontend releases separately (git tags `B-x.y.z` / `F-x.y.z`), so the two genuinely drift independently in normal operation — the `/version` page's compatibility check is the runtime signal for when that drift matters.
- `backend/release_news.json` and `frontend/public/release_news.json` carry the same independence into changelog data: `{unreleased: string[], releases: [{version, release_date, changes}]}`. `GET /api/public/release-news` reads and returns the backend's file verbatim (same pattern as `/version`); the frontend fetches its own `release_news.json` directly, since it's static, in `public/`. The `/release-news` page groups each side's `releases` by major.minor (via the same `src/utils/version.ts` helpers) and renders every generation as a two-column row — a generation with entries on *both* sides gets a "compatible" highlight, one side missing entries reads as an empty column, so (in)compatibility is visible without comparing dates.
- **Every change appends to the relevant `unreleased` array** (see Workflow requirements above). `scripts/build.py`'s `seal_release_news()` moves a component's accumulated `unreleased` entries into a new dated release under the resolved version at STABLE build time, then clears `unreleased` (a no-op, so safe to re-run, if nothing's pending). It edits the local working-tree file, not the git ref being archived — commit the sealed file and create the release tag from that commit *before* running the build, or the archived package won't include the sealed entry.

### Frontend: routing, auth flow, styling
- Routes are declared in `App.tsx` (react-router-dom) and wrapped by `AuthProvider` + `BrowserRouter` in `main.tsx`; page-level chrome (nav bar + language switcher) comes from `components/Layout.tsx`. Pages: `HomePage.tsx` (`/`, public), `Login.tsx` (`/login`, email/password + "Continue with Google"), `Register.tsx` (`/register` — registration alone issues no tokens, per the auth service's own design, so success just shows a confirmation + a link to `/login`, it does not log the user in), `Account.tsx` (`/account`, see below), `OAuthCallback.tsx` (`/oauth/callback` — extracts tokens Google's flow appends as query params), `Consent.tsx` (`/consent`), `ConsentRejected.tsx` (`/consent-rejected`), `Version.tsx` (`/version`, public diagnostic page).
- `Layout.tsx`'s nav bar is the only way to reach `/account` / `/register` / `/login` besides typing the URL directly — it swaps between Login/Register (logged out) and Account/Logout (logged in) links based on `useAuth().user`, plus a CS/EN language switcher (see i18n below).
- `api/client.ts` owns the API base URL (`VITE_API_URL`) and the JWT access/refresh token lifecycle: `apiFetch()` attaches the bearer token and transparently retries once via `/api/auth/refresh` on a 401.
- **Consent redirect is centralized in `AuthContext.tsx`**, not left to callers: both `login()` (on a 403 `consent_required` from `/api/auth/login`) and `loadUser()` (on the same shape from `/api/auth/me`) call an internal `redirectToConsent()` that navigates to `/consent?group=<id>&redirect=<target>`, preserving tokens (the user *is* validly authenticated, just missing consent — don't `clearTokens()` on this path). Four constraints here were all found by live testing against the real service, not by inspection — preserve them when touching this file:
  - `redirectToConsent`'s redirect target falls back to `/` whenever the *current* page is `/login`, `/oauth/callback`, or `/consent` itself, rather than blindly capturing `window.location.pathname`. Consent is very often detected from *within* `login()` itself (not just on page load for an already-protected route), and at that moment the "current page" is always `/login` — capturing it verbatim sends the user back to an empty login form after granting, not anywhere useful.
  - `AuthProvider`'s initial-session-restore effect (the one that calls `loadUser()` on mount if a token already exists) has an **empty dependency array (`[]`), not `[loadUser]`**, and that's intentional. `loadUser`'s identity depends on `navigate` (via `redirectToConsent`), which react-router doesn't guarantee stays referentially stable across navigations; depending on `[loadUser]` let this mount-only check silently re-fire on unrelated route changes, redundantly re-querying `/me` and, if consent was pending, re-triggering the consent redirect using whatever URL it had *just* redirected to as the new target (a doubly-nested `redirect` query param).
  - `Consent.tsx` calls `loadUser()` again right after a successful grant, before navigating away. `AuthContext`'s `user` stays `null` through the entire consent-required window (`loadUser()` returns early without setting it), so without this refresh, navigating to any page that guards on `useAuth().user` (e.g. `Account.tsx`) would immediately bounce back to `/login` on the stale `null`.
  - Both `login()` and `OAuthCallback.tsx` only navigate to `/` when `loadUser()` reports it actually loaded the user (`loadUser()` returns a `boolean`) — navigating unconditionally after a successful token exchange was a real bug (a successful OAuth login would fire `navigate('/')` right after `loadUser()` had already redirected to `/consent`, canceling it).
- Google OAuth needs no backend involvement at all: the browser is redirected straight to `${getAuthUrl()}/auth/google?redirect_uri=...`, the auth service handles the whole exchange with Google, and redirects back to `/oauth/callback` with tokens as query params directly. `VITE_AUTH_URL` is what `getAuthUrl()` reads for this — it's the one place the frontend needs to know the auth service's own public URL, since a browser redirect can't be proxied through the backend like the password-based endpoints are. (`frontend/.env.example` correctly points this at `https://auth.withfbraun.com` and no longer defines an unused `VITE_API_KEY` — both were flagged and fixed in an earlier pass; don't reintroduce either into runtime frontend code. `build.py` does write its own hardcoded `VITE_API_KEY` for a completely separate reason — see Deployment below.)
- Tailwind is v4, CSS-first: there is no `tailwind.config.js`. `src/index.css` just does `@import "tailwindcss"` plus a handful of hand-written base styles, and the `@tailwindcss/vite` plugin (in `vite.config.ts`) handles content detection automatically.
- `CORS_ORIGINS` in `backend/.env.example` (and the `Settings.cors_origins` default) match the frontend dev server's actual default port, `["http://localhost:5173"]` — keep these in sync if either default port ever changes.

### Frontend: internationalization (i18n)
- `react-i18next` + `i18next-browser-languagedetector`, initialized once in `src/i18n/index.ts`. `SUPPORTED_LANGUAGES` (currently `['cs', 'en']`) is the **single source of truth** for which languages ship — `Layout.tsx`'s language switcher iterates over it and `applyUserLanguage()` (below) checks against it; don't hardcode a separate language list anywhere else. Detection order is `localStorage` before `navigator`, with `cs` as `fallbackLng` (this template originates from a Czech-first product) — `supportedLngs` must stay pinned to the same list `resources` actually ships, or an unrecognized locale resolves to a partial/missing bundle.
- Translation keys are namespaced per page in `src/i18n/locales/{cs,en}.json` (`login.*`, `account.*`, `common.*` for shared strings, etc.) — every page component calls `useTranslation()` and reads its own strings via `t('page.key')`; there's no untranslated hardcoded copy anywhere in `src/pages/`.
- `applyUserLanguage(languageCode)` (exported from `src/i18n/index.ts`) is a fallback **cascade**, not a straight assignment: the signed-in user's own `language_code` if this app ships that language, else `'en'` if that's shipped, else **leave the current language untouched** (whatever was active before — a manual switcher choice or the detector's result). It's called from `AuthContext.tsx`'s `loadUser()` (covering session restore, login, and Google OAuth — every path that resolves who the user is) and again from `Account.tsx` after saving the Profile card, so editing `language_code` there takes effect immediately without a reload.

### Frontend: configurable auth.user attributes (`public/config.json`)
- `public/config.json` declares extra, app-specific fields to render on top of auth.user's core profile fields on the Account page's Profile card — each entry has a `key`, a `bucket` (`user_data` or `system_data`), a `type` (`text` / `number` / `boolean` / `date`), a `required` flag, and an `i18nKey`. It's fetched at runtime via `loadUserAttributes()` in `src/config/userAttributes.ts` (it's static, in `public/`, not a build-time import — editing it doesn't require a rebuild).
- **Only `bucket: "user_data"` entries ever render or save.** `system_data` entries are dropped by `loadUserAttributes()` with a `console.warn`, not silently ignored — this app's credentials can never read or write `system_data` on the auth service (see Authorization above), so a `system_data` config entry would otherwise look like a real field that silently fails to save. A config file may still declare `system_data` entries for documentation purposes; don't build a code path that tries to render one.
- `DynamicAttributeField.tsx` renders one control per attribute based on `type`. Saved values round-trip through `PATCH /api/auth/me/group-attributes/{group_id}` — since that endpoint replaces the whole `user_data` object (see Authorization above), `Account.tsx` always sends the full known set of attribute values on save, not just whichever one changed.
- `Account.tsx`'s two cards never cross endpoints: the Account card (`nickname`) only ever calls `/api/account/me`; the Profile card (core fields + the config-driven attributes above) only ever calls `/api/auth/me` and its group-attributes endpoint. Each card has its own independent saving/error state so one failing never blocks the other. One nuance worth preserving: controlled inputs store an unset field as `''` (React inputs can't hold `null`), but the auth service validates fields like `birth_date` as a real date-or-absent and rejects `""` outright — `Account.tsx`'s `blankToNull()` converts blanks to `null` right before the `PATCH /api/auth/me` request.

### File headers
Every hand-written source file (backend `.py` under `app/`, `tests/`, `alembic/`; frontend `.ts`/`.tsx`/`.css`) starts with the project header — a `#`-comment block in Python, a `/** ... */` block in TS/TSX/CSS:
```
Part of the With FBraun project template.
Author: František Braun <frantisek.braun95@gmail.com>
Freely available as a template for building custom applications.
```
It goes as the very first lines of the file — after a shebang if one exists, or before a `.d.ts` file's triple-slash reference directive, or above (never inside) an Alembic migration's auto-generated revision docstring, since Alembic's own tooling reads that docstring verbatim. Add it to new files as you create them.

### Deployment (`scripts/`)
- `build.py` archives `backend/` / `frontend/` / `scripts/` from a git ref (a `B-`/`F-` tag in STABLE mode, `HEAD` in SNAPSHOT mode), builds the frontend with `npm install && npm run build`, and uploads the resulting zip to FTP per `scripts/build.config.toml` (gitignored; copy from `.example`). For STABLE backend/frontend builds it first seals that component's `release_news.json` under the resolved version (see Cross-cutting above) — commit the sealed file and create the release tag from that commit before running the build, or the archive won't include it.
- `build_frontend()` writes its own build-time `frontend/.env` with a **deliberately public, shared trial `VITE_API_KEY`**, read from `build.config.toml`'s `client_api_key` (gitignored, like the rest of that file). It exists purely so people can try the packaged template's own live demo build without first registering their own application on the auth service. **This is not a pattern to copy for a real application** — a real app must never send its API key from the browser at all (see the BFF note under Authorization above); this key only exists because the template's own demo build has no backend-held key of its own to proxy through. Don't add a real app's `VITE_API_KEY` here, in `.env.example`, or anywhere else client-side.
- `install.sh` / `upgrade.sh` are Linux-only systemd + nginx deployment scripts (require `psql`, `nginx`, `systemctl` on `PATH`) driven by `scripts/deploy.config.sh` (gitignored; copy from `.example`). They provision the Postgres role/DB from `backend/.env`'s `DATABASE_URL`, and health-check via `GET /api/public/version` after `upgrade.sh` restarts the service — a change that breaks that endpoint will trigger `upgrade.sh`'s automatic rollback. `build_scripts()` bundles these together with the current, real `backend/.env` — unlike the backend/frontend archives (built from a git ref), this bundle always reflects the current local filesystem and is never meant to be redistributed as part of the template, only pushed to this project's own live server.
